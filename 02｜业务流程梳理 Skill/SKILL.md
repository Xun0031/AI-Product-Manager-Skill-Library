# 业务流程梳理 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 目标

把混乱、跳跃的业务描述整理为可确认的端到端流程，明确起点、终点、角色、节点输入输出、状态变化、异常分支和待确认问题。

## 2. 关键边界声明

本 Skill 只处理业务事实层，不直接设计系统模块、权限、页面、接口或数据库。业务流程经确认后，再进入产品架构与详细需求工作。

## 3. 适用场景

- 会议或访谈后还原当前流程
- SOP 与操作说明结构化
- 多角色协作、交接与异常梳理
- 流程图、架构或 PRD 的前置确认

## 4. 不适用场景

- 技术调用链或部署流程
- 详细页面交互与权限设计
- 已完全明确的流程排版

## 5. 输入材料要求

### 5.1 最小输入

提供一段流程描述、访谈摘要或操作说明。

### 5.2 推荐输入

| 输入项 | 说明 |
|---|---|
| 流程目的 | 为什么存在 |
| 起止条件 | 何时开始、何时结束 |
| 参与角色 | 谁负责什么 |
| 节点与规则 | 输入、动作、判断、输出 |
| 异常与兜底 | 退回、超时、撤回等 |

## 6. 分析原则

1. 先还原事实，再做产品转译。
2. 先主流程，再异常分支。
3. 每个节点都有输入和输出。
4. 状态来自业务事实。
5. 具体标注冲突、推断和信息缺口。

## 7. 分析流程

1. 明确流程边界。
2. 识别角色和职责。
3. 还原主流程。
4. 拆解节点输入、动作与输出。
5. 梳理状态、异常和人工兜底。
6. 识别业务数据记录点与待确认项。

## 8. 标准输出结构

输出应包含：材料理解、流程边界、角色分工、主流程、节点明细、状态变化、异常分支、数据记录点、当前问题、可产品化观察点、待确认问题、确认清单与流程自检。

## 9. 与其他 Skill 的关系

02 承接 01 的定位结果，并为 03 的角色架构、04 的方案结构和 05 的演示路径提供经确认的业务事实。

## 10. 使用注意事项

- 不把业务角色直接等同于系统账号。
- 不把业务数据记录点直接写成数据库字段。
- 分开描述当前流程与目标流程。
- 最终结果应可由业务责任人逐项确认。
