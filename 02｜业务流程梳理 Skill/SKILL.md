# 业务流程梳理 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 目标

把混乱、跳跃的业务描述整理为可确认的端到端流程，明确触发条件、输入来源、起点、终点、角色、节点责任、动作类型、状态变化、异常分支、反馈闭环、数据沉淀和待确认问题。

## 2. 关键边界声明

本 Skill 只处理业务事实层，不直接设计系统模块、权限、页面、接口或数据库。业务流程经确认后，再进入产品架构与详细需求工作。

## 3. 适用场景

- 会议或访谈后还原当前流程
- SOP 与操作说明结构化
- 多角色协作、交接与异常梳理
- 流程图、架构或 PRD 的前置确认

## 4. 不适用场景

- 技术调用链或部署流程
- 详细页面交互与权限设计
- 已完全明确的流程排版

## 5. 输入材料要求

### 5.1 最小输入

至少提供流程目的、触发方式、主要参与角色、当前已知步骤和输出用途。只提供一句任务描述时，先返回信息缺口，不直接补全流程。

### 5.2 推荐输入

| 输入项 | 说明 |
|---|---|
| 流程目的 | 为什么存在 |
| 触发与起止条件 | 谁在何种条件下发起，何时结束或重开 |
| 输入来源 | 表单、文件、消息、接口或人工补充等来源 |
| 参与角色 | 谁发起、受理、处理、审批、确认和兜底 |
| 节点与规则 | 输入、责任人、人工 / 系统动作、判断、输出 |
| 状态与时效 | 状态名称、进入条件、退出条件、超时规则 |
| 通知与审批 | 谁被通知、何时提醒、谁有权批准或退回 |
| 异常与兜底 | 缺失、重复、超时、退回、撤回、失败和人工接管 |
| 结果与数据 | 如何确认结果、向谁反馈、沉淀哪些业务记录 |

## 6. 分析原则

1. 先还原事实，再做产品转译。
2. 先主流程，再异常分支。
3. 每个节点都有输入和输出。
4. 状态来自业务事实。
5. 具体标注冲突、推断和信息缺口。
6. 每个关键节点只有一个明确的直接责任角色，可另列协作与知会角色。
7. 人工动作与系统动作分开描述，不把期望能力写成现状事实。
8. 优化建议必须对应已识别断点，并说明适用前提与影响。

## 7. 分析流程

1. 识别流程目的、触发条件、结束条件和不在范围内的事项。
2. 盘点输入来源，标注来源角色、载体、完整性要求和重复判断依据。
3. 建立角色责任表，区分发起、执行、审批、确认、协作、知会与人工兜底。
4. 还原主流程，按“输入 → 责任人 → 动作 → 判断 → 输出 → 下一状态”拆解节点。
5. 分开标注人工动作、系统动作、通知提醒和审批动作。
6. 建立状态流转表，说明进入条件、允许动作、退出条件和可否退回或重开。
7. 梳理信息缺失、重复提交、超时、退回、撤回、处理失败和结果异议等异常流程。
8. 明确结果确认、反馈对象、反馈渠道、归档条件和业务数据沉淀点。
9. 识别等待无责任人、交接无回执、状态不一致、重复录入和异常无出口等流程断点。
10. 为每个断点提出最小必要优化建议，标注收益、影响角色、前置条件和待确认风险。

## 8. 标准输出结构

输出应包含：材料理解、流程边界、触发条件、输入来源、角色责任、主流程、节点明细、人工与系统动作、状态流转、通知提醒与审批、异常分支、结果反馈、数据沉淀、流程断点、优化建议、待确认问题、业务确认清单与流程自检。

节点明细至少使用以下颗粒度：

| 节点 | 触发 / 输入 | 直接责任人 | 人工动作 | 系统动作 | 判断与规则 | 输出 | 状态变化 | 通知 / 审批 | 异常出口 | 数据沉淀 |
|---|---|---|---|---|---|---|---|---|---|---|
| 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 |

## 9. 与其他 Skill 的关系

02 承接 01 的定位结果，并为 03 的角色架构、04 的方案结构和 05 的演示路径提供经确认的业务事实。

## 10. 使用注意事项

- 不把业务角色直接等同于系统账号。
- 不把业务数据记录点直接写成数据库字段。
- 分开描述当前流程与目标流程。
- 不用“相关人员”“后台处理”等模糊表述代替责任角色。
- 无法确认时效、审批权或系统自动化现状时，明确标记为待确认。
- 最终结果应可由业务责任人逐项确认。
