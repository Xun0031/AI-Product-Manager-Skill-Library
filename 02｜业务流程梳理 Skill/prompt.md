# 02｜业务流程梳理 Skill｜Prompt

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

使用时只填写匿名化后的必要信息。未知内容可留空，并要求 AI 标记为待确认。

复杂流程优先使用下面的标准模板；后续场景模板可作为补充约束，不应删除责任、状态、异常和数据字段。

## 1. 标准调用提示词

### 使用场景

完整执行本 Skill 的标准分析流程，并对信息缺口单独标注。

### 输入

```text
【流程目的与输出用途】
{{填写流程为什么存在，以及结果用于确认、评审、SOP 还是产品设计前置}}

【触发条件与结束条件】
{{填写由谁、在何种条件下发起；何时完成、取消、归档或重开}}

【输入来源】
{{填写表单、文件、消息、接口或人工补充等来源；说明完整性和重复判断线索}}

【参与角色与已知分工】
{{填写发起、受理、处理、审批、确认、知会和兜底角色；未知可留空}}

【已知主流程与异常】
{{填写当前步骤，以及信息缺失、重复、超时、退回、失败、异议等情况}}

【状态、通知与记录】
{{填写已知状态、提醒、审批、反馈、归档和数据记录要求}}

【约束与禁止事项】
{{填写时间、范围、表达边界和不能出现的内容}}
```

### 执行要求

1. 先区分已确认事实、合理推断和待确认项，冲突信息单列。
2. 先画定触发、起点、终点和重开边界，再拆主流程与异常流程。
3. 对每个节点列出输入、直接责任人、人工动作、系统动作、判断、输出和下一状态。
4. 单独整理状态流转、通知提醒、审批、结果确认、反馈和数据沉淀。
5. 识别无责任人、无回执、无异常出口、重复录入、状态不一致和人工兜底缺失等断点。
6. 优化建议逐条对应断点，说明前提、影响角色和待确认风险。
7. 信息不足时先给初步结构，再提出能由具体责任角色回答的问题。
8. 不编造数据、结果、主体、时效、规则、政策依据或系统能力。

### 输出要求

- 使用清晰的 Markdown 标题、列表和必要表格。
- 先给结论，再给依据与细节。
- 主流程与异常流程分开表达。
- 输出角色责任表、节点明细表、状态流转表、断点与优化建议表。
- 明确列出风险、待确认项和下一步。
- 结尾执行对应 checklist 自检。

## 1.1 复杂多方协作补充提示词

### 使用场景

流程存在多来源输入、跨角色交接、补充材料、重复提交、超时、退回重开、结果确认或人工兜底时使用。

### 补充执行要求

1. 为每类输入定义来源、接收节点、完整性检查和重复识别方式。
2. 对每次交接写明交出方、接收方、交接物、回执和超时责任。
3. 将“补充后继续”与“退回后重开”区分为不同状态和路径。
4. 明确自动提醒升级到人工跟进的条件，但未知阈值不得自行设定。
5. 结果确认失败时，说明复核、异议或人工接管路径。
6. 给出一份可直接用于业务访谈复核的确认问题清单。
