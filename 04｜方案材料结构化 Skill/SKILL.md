# 方案材料结构化 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 目标

把产品分析结果转译为可沟通、可汇报、可评审的材料结构，明确汇报对象、材料目标、事实依据、推断与建议、章节主线、页级观点、图表建议、讲述重点、信息缺口和表达边界。

## 2. 关键边界声明

本 Skill 负责材料结构设计，不直接制作演示文稿、视觉稿、详细 PRD、技术方案或代码。

## 3. 使用前提

- 系统定位基本明确
- 业务流程和产品架构已有可引用结论
- 沟通对象、场景与预期动作可描述

## 4. 适用场景

- 方案沟通与决策汇报
- 立项、评审或阶段说明
- Demo 配套讲解
- 多方协作方案的结构化表达

## 5. 不适用场景

- 前置分析尚不清楚
- 直接进行视觉制作
- 详细需求或技术设计
- 无依据的包装性表达

## 6. 输入材料要求

### 6.1 最小输入

至少提供汇报对象、材料目标、使用场景、已有事实和期望对方采取的动作。只提供项目名称或一句“做个方案”时，先返回信息缺口，不直接生成目录。

### 6.2 推荐输入

| 输入项 | 说明 |
|---|---|
| 汇报对象与场景 | 谁看、在什么场合使用、对方已知什么 |
| 材料目标与预期动作 | 希望对方理解、确认、决定、执行或对外转述什么 |
| 前置结论 | 定位、流程、架构与边界 |
| 事实依据 | 可验证的信息、来源类型和确认状态 |
| 版本边界 | 对内 / 对外，高层决策 / 业务执行 |
| 制作约束 | 时长、载体、页数、风格和禁止表达 |

### 6.3 可选补充信息

可提供已有目录、匿名草稿、图表思路或 Demo 规划。

## 7. 分析原则

1. 先对象后结构，先目标后内容，先主线后材料页。
2. 先问题和价值后能力，一页只表达一个核心观点。
3. 将内容标记为事实、推断、建议或待确认，不用建议伪装事实。
4. 对内材料可呈现方案取舍、责任、依赖和执行风险；对外材料只保留已授权、可验证和可公开表达的内容。
5. 高层决策版突出决策事项、价值、关键取舍和风险；业务执行版突出范围、流程、分工、计划和验收；对外沟通版突出共同目标、协作方式、已确认能力和边界。
6. 图表建议必须服务于核心观点，不能用装饰性图表替代证据。

## 8. 分析流程

1. 识别汇报对象、角色层级、决策责任、已知信息和关注点。
2. 判断材料目标与预期动作，区分决策、评审、执行、说明和对外沟通。
3. 盘点输入，按事实、推断、建议、待确认四类标记，并记录可引用依据。
4. 判断对内 / 对外边界和高层 / 执行深度，列出禁止出现或需授权的内容。
5. 提炼一句话主线，设计“为什么 → 做什么 → 如何做 → 如何判断 → 风险与动作”的章节逻辑。
6. 设计页级结构，为每页写明核心观点、支撑事实、图表建议、讲述重点和表达边界。
7. 识别缺少证据、责任、范围、计划或验收标准等信息缺口，指定确认对象。
8. 配置正文、附录、Demo 嵌入点和不同版本之间的保留、删减与改写关系。
9. 输出风险、边界、待确认事项和后续制作建议，并执行方案表达自检。

## 9. 标准输出结构

输出应包含：汇报对象、材料目标、预期动作、事实 / 推断 / 建议分层、内外边界、版本策略、方案主线、章节目录、页级结构、页面核心观点、支撑事实、图表建议、讲述重点、表达边界、Demo 嵌入、正文与附录、信息缺口、风险、待确认事项和后续制作建议。

页级结构至少使用以下颗粒度：

| 页序 / 章节 | 页面目的 | 核心观点 | 支撑事实 | 建议图表 | 讲述重点 | 表达边界 | 信息缺口 |
|---|---|---|---|---|---|---|---|
| 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 | 【填写】 |

## 10. 与其他 Skill 的关系

04 引用 01 至 03 的结论形成材料结构，并为 05 的 Demo 规划提供讲述主线；成品材料可由 06 质检。

## 11. 使用注意事项

- 不重新进行前置分析。
- 不编造政策、成果或量化效果。
- 不让功能列表取代问题与价值主线。
- 不把内部假设、未定责任、执行风险或未授权信息直接迁移到对外版本。
- 页面没有支撑事实时，标记为待补证或删除，不用空泛口号填充。
- 重要表达需由责任人复核。
