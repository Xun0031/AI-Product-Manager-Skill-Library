# 系统定位梳理 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 目标

从零散输入中判断系统本质、主要服务对象、关键问题、核心价值与建设边界。先回答“系统是什么”，再讨论“系统做什么”，并区分已确认事实、合理推断与待确认项。

## 2. 适用场景

- 新系统启动前统一方向
- 多方诉求较多但产品定义不清
- 方案、架构、Demo 或 PRD 的前置分析
- 需要面向不同受众形成一致表达

## 3. 不适用场景

- 定位已经明确，只需详细需求设计
- 单点文案、页面或技术方案
- 接口、数据库或部署架构设计

## 4. 输入材料要求

### 4.1 最小输入

至少提供背景说明、访谈摘要、需求描述、问题清单或方案草稿中的一种。

### 4.2 推荐输入

| 输入项 | 说明 |
|---|---|
| 建设背景 | 为什么需要系统 |
| 参与角色 | 谁使用、协作或查看结果 |
| 当前问题 | 现有方式的主要困难 |
| 目标与约束 | 希望达成什么、有哪些限制 |
| 输出用途 | 用于讨论、方案、架构或演示 |

## 5. 分析原则

1. 先定位，后功能。
2. 区分诉求、问题、目标和能力。
3. 明确系统不是什么。
4. 价值表达不止于功能。
5. 不为追求完整而补造事实。

## 6. 分析流程

1. 识别输入类型并提取事实。
2. 拆分角色、问题、目标、诉求和约束。
3. 判断系统本质与主服务对象。
4. 提炼价值主线和关键场景。
5. 划定首期、后续与暂不支持范围。
6. 输出分受众表达和待确认问题。

## 7. 标准输出结构

输出应包含：输入理解、关键信息拆解、角色关系、系统本质、系统不是什么、核心场景、关键问题、价值主线、系统边界、一句话定位、分受众表达、待确认问题与定位自检。

## 8. 使用注意事项

- 不把功能清单当成系统定位。
- 避免空泛的平台化表达。
- 输入不足时输出初步判断并列出缺口。
- 定位结论需由相关责任人确认。

## 9. 与其他 Skill 的关系

01 的输出为 02 的流程梳理、03 的产品架构、04 的方案结构和 05 的 Demo 规划提供前置方向；它不替代这些后续工作。
