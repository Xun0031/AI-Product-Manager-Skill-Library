# 角色驱动的产品架构梳理 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 目标

把业务角色转译为产品侧系统角色，并统一梳理工作台、数据可见范围、关键操作、页面范围和协作关系。

## 2. 关键边界声明

本 Skill 讨论产品架构，不是技术架构；不输出数据库、接口、服务部署、详细交互或字段级权限。

## 3. 使用前提

- 系统定位与价值边界基本明确
- 业务流程、角色和关键状态已完成初步确认
- 已知首期目标和主要约束

## 4. 适用场景

- 多角色 B 端平台
- 工作台与端口规划
- 数据和操作边界前置分析
- PRD、方案或 Demo 前的架构梳理

## 5. 不适用场景

- 定位或流程尚不清楚
- 单角色简单工具
- 技术架构、数据库或按钮级权限设计

## 6. 输入材料要求

### 6.1 最小输入

提供定位结果、流程说明或角色清单中的一种。

### 6.2 推荐输入

| 输入项 | 说明 |
|---|---|
| 业务角色 | 职责与使用频率 |
| 核心数据 | 数据对象与归属 |
| 关键动作 | 角色需完成的业务动作 |
| 协作关系 | 谁向谁交接什么 |
| 首期目标 | 必须支持的闭环 |

## 7. 分析原则

先业务角色后系统角色；先职责后权限；先数据范围后页面范围；先主角色后辅助角色；先首期后扩展。

## 8. 独立端口判断标准

| 判断条件 | 说明 |
|---|---|
| 高频使用 | 经常登录和操作 |
| 独立任务 | 有稳定、独立的任务 |
| 独立数据范围 | 只查看特定范围 |
| 直接影响结果 | 操作改变流程状态 |

低频、只被记录或规则未定的角色，优先作为数据对象、代操作对象或后续扩展。

## 9. 分析流程

1. 提取业务角色。
2. 判断是否转为系统角色。
3. 划分主辅角色与工作台。
4. 定义职责、数据范围和操作边界。
5. 规划页面范围与协作关系。
6. 区分首期、后续和暂不支持。

## 10. 标准输出结构

输出应包含：角色总览、转换判断、角色分层、独立端口判断、工作台划分、职责、数据范围、操作边界、页面范围、协作关系、阶段范围、风险和待确认问题。

## 11. 与其他 Skill 的关系

03 使用 01 和 02 的结论形成产品架构基础，并向 04 提供方案表达素材、向 05 提供 Demo 的角色与页面范围。

## 12. 使用注意事项

- 不把沟通载体当成系统角色。
- 不默认每个角色都需要独立端口。
- 不机械复制页面。
- 涉及敏感数据时由安全与合规责任人复核。
