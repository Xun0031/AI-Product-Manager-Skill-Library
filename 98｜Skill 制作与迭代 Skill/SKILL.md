# 98｜Skill 制作与迭代 Skill

> 使用前请先对输入信息进行匿名化处理，不要粘贴客户机密、个人隐私、内部链接、账号密钥、未授权的项目原文或真实业务数据。示例中的组织、人物、项目和数据均为虚构或脱敏内容。

## 1. Skill 定位

这是用于设计和迭代 Skill 的 Meta Skill。它先判断是否值得沉淀，再设计边界、结构、测试和版本策略。

## 2. 与业务 Skill 的关系

98 不直接替代 01 至 07 完成业务任务，只负责 Skill 化判断、结构设计、文件检查和迭代诊断。

## 3. 默认产出边界

默认输出判断、结构设计稿或检查报告；未获得明确授权时，不创建、修改、移动、发布文件。

## 4. 核心价值

- 防止一次性任务被过度工程化
- 优先复用和扩展已有 Skill
- 把经验转化为可调用、可测试、可迭代的资产

## 5. 适用场景

- 判断想法是否值得 Skill 化
- 设计新 Skill 结构
- 检查现有 Skill 文件
- 根据使用问题进行迭代诊断

## 6. 不适用场景

- 直接处理具体业务任务
- 将所有提示词都升级为 Skill
- 未经测试直接标记稳定版本
- 未授权执行文件变更

## 7. 五种工作模式

1. Skill 化价值判断。
2. Skill 结构设计。
3. 结构设计稿自检。
4. 文件结构检查。
5. Skill 迭代诊断。

## 8. 输出强度

轻量模式只给结论、理由、重叠关系和下一步；完整模式才展开定位、边界、流程、文件、测试和版本建议。

## 9. 输入材料

可提供场景、用户、频率、现有提示、典型输入输出、质量标准、已有 Skill、测试结果和本次授权范围。

## 10. 判断标准

| 维度 | 判断问题 |
|---|---|
| 高频性 | 是否会反复出现 |
| 复杂性 | 是否需要稳定流程 |
| 输入输出稳定性 | 是否可形成模板 |
| 边界 | 能否说清不做什么 |
| 质量标准 | 能否判断好坏 |
| 可测试性 | 能否设计正反案例 |

## 11. 推荐处理方式

A：独立 Skill；B：轻量提示模板；C：扩展已有 Skill；D：沉淀为案例或检查项；E：暂不沉淀。

## 12. 核心原则

先场景后 Skill；先判断后设计；优先复用；先边界后模板；先结构稿后文件；先测试后稳定版本。

## 13. 标准工作流程

理解想法 → 判断模式 → 评估价值 → 检查重叠 → 推荐方式 → 定义定位与边界 → 设计输入输出与流程 → 设计测试 → 给出阶段结论。

## 14. candidate 判定门槛

候选状态至少要求正式文件完整、结构检查通过、正向/边界/反向测试完成、问题已收敛、版本记录已更新。候选状态不等于长期稳定版本。

## 15. test-results 边界

测试目录只保留合成输入、测试过程和结论；不存放未授权材料、个人信息或完整业务交付物。
